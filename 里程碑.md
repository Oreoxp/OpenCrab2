里程碑 1：Codex 基座打通

这一阶段只做一件事：

让 Codex 从你的 UI 里真正跑起来。

要做什么
Rust core 能拉起/管理 Codex 进程
能选择一个本地项目目录作为 workspace
UI 能发 prompt 给 Codex
能流式接收运行状态
能看到：
思考中
读文件
改文件
跑命令
完成/失败
能停止任务、重新开始任务、保存任务记录
这一阶段先不要做
美观 UI
技能系统
复杂 diff
自动化
多窗口
验收标准

你能在一个真实 repo 上完成这个动作：

在 UI 中打开项目 → 输入“帮我给某个函数补日志并加测试” → Codex 开始执行 → UI 能持续显示过程 → 最终返回结果

只要这个打通，你们就真正有“基座”了。

里程碑 2：核心工作台成型

这一阶段是把“能跑”升级成“能用”。

必须有的 6 个面板/区域
项目区
当前 workspace、分支、任务列表
会话区
用户输入、Agent 回复、状态流
操作流区
Agent 当前读了什么、改了什么、跑了什么命令
审批区
对危险动作进行确认
例如：
执行命令
删除文件
Git reset / checkout / commit
网络访问
Diff 区
查看修改内容
接受/拒绝当前修改
日志区
runtime 日志
错误原因
命令输出
这一阶段最重要的产品点

不是“好看”，而是可理解。
用户必须知道：

Agent 正在干什么
为什么卡住
它想让我批什么
改了哪些文件
我怎么回滚
验收标准

你可以完整走完一条链路：

提需求 → Agent 执行 → 用户审批一个命令 → Agent 改代码 → UI 展示 diff → 用户决定接受还是回退

做到这里，这个产品就已经不是 demo 了。

里程碑 3：安全与可控闭环

这是你们产品以后能不能立住的关键。

要做什么
权限模型
只读模式
可编辑模式
命令需审批模式
高风险命令黑名单/白名单
工作区隔离
限制只能操作当前项目目录
明确不可触碰目录
为后续 worktree / sandbox 留接口
审计记录
记录：
prompt
文件读取
文件修改
命令执行
审批历史
最终结果
失败恢复
中断后恢复会话
上一轮 diff 可找回
错误原因可查看
为什么这个里程碑这么重要

因为从这一步开始，你做的就不是一个“聊天壳子”，而是一个可信执行工作台。

验收标准

你故意制造几种风险场景，产品都能稳住：

Agent 想删文件
Agent 想跑危险命令
Agent 中途崩溃
Agent 输出异常
用户拒绝审批

系统都能给出明确结果，而不是“糊成一团”。

里程碑 4：Skills 体系落地

这一步决定你们以后能不能形成自己的护城河。

第一版 skills 不要设计太重

我建议先做成：

一个 skill = 一份元数据 + 一段说明 + 可选脚本/模板
可以本地导入/导出
可以在任务里显式调用
可以附加到某类任务默认启用
第一批 skill 示例

建议直接做 3 个：

代码审查 skill
按你的团队风格输出 review
聚焦风险、测试、兼容性
Bug 排查 skill
自动收集关键文件
给出排查路径
优先最小修改
提交整理 skill
帮你生成 commit message
总结变更点
归纳影响范围
这一阶段的关键

不是 skill 数量，而是：

skill 是否真的能复用。

验收标准

同一个 skill 在至少 3 个不同 repo / 任务里都能复用，而且输出风格基本一致。

里程碑 5：MVP 封版与对外试用

做到这里，才算第一版真正成立。

要做什么
打磨安装和启动流程
加基础设置页
做最小 onboarding
准备 3 个标准演示场景
修一轮最影响体验的 bug
你们对外展示时，应该只讲 3 个卖点

不要讲太多，第一版就讲：

本地仓库可控执行
审批 + diff + 回滚
skills 复用
