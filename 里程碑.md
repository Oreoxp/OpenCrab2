小螃蟹项目里程碑（基于 Codex，先能力验证、后产品壳、再做登录与扩展）
总体原则

小螃蟹不从 0 开发引擎，而是基于 Codex 现有 CLI / app-server / MCP / 本地执行能力，先做一个可运行、可交互、可验证的产品壳。

当前阶段优先级如下：

先确认 Codex CLI 现有能力边界
先把本地可运行版本编译出来
先做产品壳与 UI 原型
先接已有 MCP / 本地能力
后做 provider / 登录改造
soul / 记忆后置，不作为 v0.1 核心目标

项目目标不是“深改 Codex 内核”，而是“尽量复用 Codex runtime，把小螃蟹做成一个聊天驱动的本地助手产品壳”。

里程碑 0：运行验证与能力盘点
目标

先把 Codex 仓库编译运行起来，并建立一份“现有能力清单”，确认哪些功能已经存在，哪些只需要 UI 接入，哪些真的需要改代码。

任务
编译并运行 Codex 本地仓库
实际跑通以下入口：
CLI 主入口
codex app
codex app-server
codex mcp
codex mcp-server
梳理现有命令、配置项、运行模式、日志输出
确认以下能力是否已内建：
MCP 管理
本地命令执行
文件读写
多线程 / 会话
app-server 协议通信
审批 / 权限 / sandbox
产出一份能力矩阵，按三类标记：
已具备，只需 UI 接入
已具备，但要做中间层适配
尚不具备，需要新增开发
验收标准
本地能稳定启动 Codex
能实际演示至少 3 条典型流程
形成一份“能力盘点表”
明确 v0.1 不需要自己重做的部分
里程碑 1：小螃蟹产品壳 v0（先壳后改底）
目标

先做一个“小螃蟹”的最小产品壳，风格参考 Codex app，但不追求完整复刻，只保留最有价值的桌面交互骨架。

任务
做一个独立 UI 壳，命名为“小螃蟹”
UI 优先模拟这些核心区域：
左侧项目/线程列表
中间聊天区
底部输入区
右侧任务/步骤/工具状态区
UI 不直接连 core，统一通过 app-server 或 host adapter 通信
优先支持这些交互：
新建线程
发送消息
展示 agent 执行步骤
展示工具调用结果
展示错误与重试
展示审批/确认动作
不做
不先做复杂设置页
不先做完整桌面 App 细节
不先做 provider 改造
不先做记忆系统
验收标准
能像一个“桌面聊天助手”一样基本使用
能发起一次任务并看到执行过程
能在 UI 中完成最小闭环，而不只是调用 CLI 文本输出
里程碑 2：Host Adapter / 中间适配层 v0
目标

建立小螃蟹自己的薄中间层，统一承接 UI、桌面端、未来 IDE 端与 Codex app-server 的交互。

任务
定义“小螃蟹 host adapter”
统一封装这些能力：
thread 创建与切换
turn 发起
steer / 追加指令
流式事件转 UI 事件
工具调用事件转展示模型
审批事件转 UI 操作
约定前端只依赖 host adapter，不直接依赖 Codex 内部事件细节
设计一层小螃蟹自己的 view model，屏蔽底层协议变化
验收标准
UI 与 Codex 之间已经不再直接耦合
后续更换 provider / 增加记忆 / 增加 MCP 面板时，不需要大改 UI 主逻辑
里程碑 3：本地能力与 MCP 接入 v0
目标

把“已经存在于 CLI 的能力”接到小螃蟹 UI 里，让它先像一个真正的本地助手。

任务
接入 MCP 管理能力
在 UI 中展示 MCP server 状态
支持查看、启停、切换常用 MCP
在 UI 中暴露本地执行能力：
当前工作目录
文件读写
命令执行
工具运行结果
增加项目 / workspace 切换
增加基础权限确认体验
重点

优先接“CLI 已经有”的功能，不优先新增复杂能力。

验收标准
用户能在小螃蟹 UI 中看到并使用 MCP
用户能明确看到本地执行发生了什么
小螃蟹已经具备“聊天 + 本地 + MCP”的基本形态
里程碑 4：Provider / 登录改造 v0
目标

在产品壳和中间层稳定之后，再把默认登录方式从 OpenAI 中心化改成 Provider 中心化。

任务
抽象 provider registry
支持至少以下 provider 类型：
OpenAI API Key
OpenAI-compatible API
本地模型 provider
将 ChatGPT 登录降级为兼容选项，而不是唯一主入口
增加 provider 选择与配置 UI
增加模型能力标记
是否支持工具调用
是否支持长上下文
是否适合本地执行
设计 provider 级 auth provider 接口
不做
不在这个阶段重写 thread / turn 主循环
不深改 core 事件结构
验收标准
小螃蟹不再依赖单一登录路径
用户可以通过添加 API / 本地模型来使用小螃蟹
登录改造不会破坏现有 UI 壳与 MCP 接入

目标

把前面拼起来的能力做成“可演示、可测试、可继续迭代”的第一版产品。

任务
设置页
Provider 管理页
MCP 管理页
工作区管理
基础日志与错误面板
崩溃恢复 / 会话恢复
基础安装与启动说明
最小品牌化：
名称

默认文案
空状态引导

可以清楚理解小螃蟹能做什么
团队可以在此基础上继续加功能，而不是推倒重来
后续里程碑（后置，不进入 v0.1 核心）

里程碑 7：soul / persona 实验版

这一步更后置。

内容包括：

soul.md

自定义默认协作风格
当前阶段的产品判断

小螃蟹 v0.1 的核心，不是“更聪明”，而是：

先能跑
先能看
先能接本地能力
先能接 MCP
先有一个像产品的壳

所以当前最优先的不是 soul，也不是完整记忆，而是：

运行验证 → 能力盘点 → 产品壳 → 中间层 → MCP / 本地能力 → provider / 登录

实施建议

请先按这个顺序推进：

第一步

先把 Codex 编译出来并跑通主要入口，产出能力盘点表。

第二步

确认哪些 CLI 能力已经内建，尤其是：

MCP
app-server
app
本地执行
配置共享
第三步

开始做“小螃蟹 UI 壳 v0”，不要先深改内核。

第四步

待 UI 壳跑通后，再推进 provider / 登录改造。

第五步

记忆和 soul 统一后置。
